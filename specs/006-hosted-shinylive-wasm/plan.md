# Feature Plan: Hosted Shinylive WebAssembly App

**Feature dir**: `specs/006-hosted-shinylive-wasm`
**Date**: 2026-07-13
**Review budget**: Keep this file under 100 nonblank lines.

## Goal

- Build and deploy this repo's public GitHub Pages app as a reproducible Shinylive/WebAssembly artifact; the existing `openanalysis.org` deployment remains separate during migration.
- Preserve package and app behavior by pinning `OpenSpecy`, app dependencies, small libraries, generated assets, and deployment metadata to the app release.

## Scope

- **In**: Review the current companion Shinylive repo for parity, fix the wasm CRAN-like repo workflow, generate a Shinylive app from `inst/shiny/`, stage medoid/model libraries, pin package/dependency versions, and add CI/manual smoke verification.
- **Out**: Rewriting the Shiny UI, changing local `run_app()` behavior beyond wasm compatibility fixes, bundling generated Shinylive site files into the source package, and supporting full derivative/nobaseline/raw libraries in WebAssembly.
- **Users**: Web users of OpenSpecy, maintainers publishing package/app releases, and local users who should see the same app except for WebAssembly library-size limits.

## Requirements

- R1. `.github/workflows/deploy-cran-repo.yml` builds a CRAN-like wasm repository containing `OpenSpecy` from the checked-out source plus the non-base app runtime dependency closure.
- R2. The wasm repository records package names, versions, and build identifiers through `PACKAGES`, lock/manifest metadata, or equivalent generated files.
- R3. The Shinylive app is generated from `inst/shiny/` and never requires hand edits to generated site or wasm repository artifacts.
- R4. The hosted app embeds the package/dependency library image built for its exact commit and does not resolve required packages from external repositories at runtime.
- R5. GitHub Actions stages only `medoid_derivative`, `medoid_nobaseline`, `model_derivative`, and `model_nobaseline` libraries with guarded `get_lib()` downloads and size reporting.
- R6. WebAssembly library selection exposes only medoid/model choices; local bundled Shiny may continue exposing full libraries.
- R7. Browser or CI-guarded smoke tests verify app startup, static assets, dependency resolution, library availability, and at least one identification/library-matching path.
- R8. Documentation/NEWS describe the hosted app release model, local-vs-WebAssembly library difference, and dependency pinning policy.
- R9. One native GitHub Pages deployment publishes pkgdown at the site root and the self-contained app at `/openspecy/`; build workflows never push generated files to `gh-pages`.
- R10. The pkgdown homepage embeds `/openspecy/`, reports real Shiny readiness while documentation remains browsable, and offers a full-screen app view with a direct-page fallback.

## Technical Decisions

- **Approach**: Build the complete wasm repository as a pinned Actions artifact, embed its library image into Shinylive, assemble pkgdown plus `/openspecy/`, embed that route on the homepage with a same-origin readiness handshake, and publish the tree once with native Pages actions.
- **Public API**: No new exported R API by default; any wasm-specific app branching should be internal app configuration or generated build metadata.
- **Dependencies**: App runtime packages detected from `DESCRIPTION` and `inst/shiny/global.R` include `shiny`, `shinyjs`, `shinyWidgets`, `bs4Dash`, `dplyr`, `DT`, `ggplot2`, `reshape2`, `plotly`, and OpenSpecy imports; final closure should be generated by pak/renv/webr tooling.
- **OpenSpecy contract**: Library staging and smoke tests must load real `OpenSpecy` libraries and exercise `load_lib()`/`match_spec()` without altering `wavenumber`, `spectra`, `metadata`, IDs, or object attributes.
- **Generated artifacts**: Generated Shinylive site, wasm package repo contents, `PACKAGES`, lock/manifest, and library staging outputs are deployment artifacts; commit only source workflow/config unless an explicit release artifact path is chosen.
- **External resources**: Use the Moore-Institute companion repo as transition/parity context; GitHub Actions may use r-wasm/shinylive actions, CRAN/r-universe/r-wasm package sources, OSF/CloudFront library URLs used by `get_lib()`, and GitHub Pages; all network paths need guarded failure messages.
- **Reference workflow compatibility**: No full reference-library rebuild; staged small-library artifacts must preserve library type names, wavenumber axes, metadata names, and representative matching behavior.
- **Bundled Shiny app**: `inst/shiny/` remains the canonical app source; only add wasm-aware configuration needed to restrict libraries and use staged local library assets in Shinylive.
- **Hosted Shinylive/WebAssembly app**: Build from `inst/shiny/`, record pinned artifact/package metadata, stage medoid/model libraries, embed the wasm image, publish generated app outside the source package, and smoke-test startup plus matching.

## Package Surfaces

- `R/`: Change only if app/library helpers need offline-safe path or wasm-environment detection.
- `tests/testthat/`: Add focused tests for app dependency manifest helpers, wasm library allow-list, and staged library path behavior if implemented in R.
- `benchmarks/`: N/A - deployment feature, no same-output package function improvement planned.
- `workflows/`: Add library staging/build helper scripts if easier than inline YAML; keep generated outputs out of source.
- `.github/workflows/`: Fix `deploy-cran-repo.yml`; add Shinylive build/deploy workflow; optionally retire or repurpose `release-file-system-image.yml`.
- `inst/`: Add minimal wasm app config/assets only if needed; do not bundle generated Shinylive output.
- `vignettes/README/pkgdown`: Update source docs for hosted app, webR/Shinylive install model, and small-library limitation.
- `pkgdown/`: Add source CSS/JavaScript for the responsive app frame, loading state, readiness handling, and full-screen control.
- `DESCRIPTION`: Add `Suggests` for build/test tooling only if repo tests call it; keep action-only tools out of package metadata.
- `NEWS.md`: Add hosted Shinylive/WebAssembly app and pinned wasm repo release note.
- Bundled Shiny app: Restrict web build through config, not by removing full-library local behavior.
- Hosted Shinylive/WebAssembly app: Required workflow, package/dependency pins, dependency closure manifest, medoid/model libraries, generated deployment artifact, smoke test.

## Risks And Open Questions

- The current r-wasm reusable workflow may not expose enough control for full dependency closure and lock metadata; may need a custom workflow around pak/webr.
- Public Pages publication remains unverified until the maintainer pushes the native deployment workflow; repository **Source: GitHub Actions** is already configured.
- Some app dependencies may lack wasm binaries or require source patches; identify blockers before changing app behavior.
